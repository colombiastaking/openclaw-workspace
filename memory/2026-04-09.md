### Network Explorer — IP-Based Node Clustering + DB Integration

**Problem identified:** Most nodes don't appear as peers in the observer's peerinfo → no IP data for them → no clustering → all non-peer nodes scatter randomly → chaotic graph.

**Goal:** Every node should know its IP(s), so nodes at the same physical server/VPS cluster together visually regardless of whether they appear as peers.

**Implementation (2-part fix):**

**1. collector.js — IP enrichment from DB peer_observations:**
- Added `getFullIPRegistry()` method to `server/db.js` — returns ALL rows from `peer_observations` table (not just latest)
- Collector now builds `pid→ips[]` from 176 prior collector runs (151K+ observations)
- Adds `n.ips = [...new Set(pidToIPs[pid])]` to every node with a known PID
- Also builds `ipToNodes{}` map: IP → all BLS keys that have used that IP
- Stats after re-run: **1,301 nodes with IP data** (95% coverage), **687 unique IPs**, **7,518 edges**
- Output now includes `ipToNodes{}` and `ipNodes{}` for app.js

**2. app.js — Pre-positioning before ForceAtlas2:**
- Rewrote IP clustering section to use `n.ips[]` directly on each node (was using edge-inference which only covered peer nodes)
- Groups nodes sharing the same IP into tight visual clusters (grid layout, 8px jitter)
- Nodes at same physical host appear as a tight group before physics settles them
- LAN IPs (192.168.x.x, 10.x.x.x, 172.16-31.x.x) excluded from clustering

**3. .gitignore — DB files excluded:**
- `data/network-explorer.db` and `data/network-explorer.db-shm` added to `.gitignore`
- DB was 115 MB — too large for GitHub, and already persists independently on the Pi

**Results after collector re-run:**
