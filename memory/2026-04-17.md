# 2026-04-17 — Network Explorer STEP F Bug

## Problem
STEP F not assigning real IPs to procryptostaking and other research-db providers. Despite `152.53.37.147` being in research-db.json, procryptostaking retained `10.254.0.1` with only STEP E edges after collector runs.

## Root Cause Found
MARKER1 debug logging was placed OUTSIDE the if/else block — firing for EVERY provider iteration, not just procryptostaking. This caused confusion in log interpretation:

```javascript
if (resolvedIP && !ipTaken.has(resolvedIP)) {
    bestIP = resolvedIP;
    bestScore = 100;
    console.log('ZZZZZZZZZ MARKER1 bestIP=' + bestIP + ...); // ← fires for ALL
} else {
    // scoring loop...
}
// MARKER1 here was INSIDE the if/else above, logging after each branch
```

## STEP F Bug — FIXED (15:57)

Two bugs found and fixed in `collector.js`:

### Bug 1 — `confidenceTier` scope
- `let confidenceTier = 'low'` was declared INSIDE the `else` block (line ~1398)
- Research path set `bestIP` and `bestScore = 100` but never entered the `else` block
- After the if/else, `confidenceTier` was referenced → `ReferenceError` (silent crash for research path)
- Fix: declared `let confidenceTier = 'low'` BEFORE the if/else; set `confidenceTier = 'high'` in research path (since researchIP has high confidence)

### Bug 2 — IP assignment block inside `else`
- The entire `if (bestIP) { ipTaken.add(bestIP); assign to members }` block was NESTED inside the `else` block
- Research path set `bestIP` but then exited the if/else — the IP assignment block was inside `else`, never executed
- Fix: moved the IP assignment block AFTER the if/else closes (both paths now execute it)
