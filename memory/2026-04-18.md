## Network Explorer — Moonlorian Geo Fix (STEP 6)

### Problem
Moonlorian nodes (14 nodes, IP `88.99.253.20`) were in `network-data.json` but had no geo coordinates.
Root cause: `getUncachedPublicIPs()` only queries `peer_observations` table (IPs seen by your 7 observers via P2P).
Since moonlorian's IP never appeared in any observer's peer list, it was never queued for geo lookup.

### Fix Applied
Added **STEP 6** to `server/collector.js` — after `buildOutput` writes `network-data.json`, do a second geo pass:
- Scan ALL unique public IPs from `network-data.json` (not just `peer_observations`)
- Geo-locate up to 1 new IP per run via ip-api.com
- Update `geo_cache` and `network-data.json` with results

**Commit:** `4529e47` (same as IPINFO_TOKEN env var fix — already pushed)

### Code Changes
- Extracted `fetchAndStoreGeo` to module level (callable from both Step 3.5 and Step 6)
- Added `step6_GeoFromNetworkData()` function
- Added `getUncachedPublicIPsFromOutput()` function
- Both functions are small and fast (SQLite queries, no external API calls)

### How It Works
- Step 3.5: Geolocates IPs from `peer_observations` (observer-seen P2P connections) — up to 5/run
- Step 6: Catches the rest — IPs that only appeared via MultiversX API (e.g. moonlorian) — up to 1/run
- Rate limit: ip-api.com allows 45k req/day; 1 per run is conservative

```
   Fetching geo for: 88.99.253.20

   Updated 14 nodes with geo for 88.99.253.20
```

### Other IPs That May Need Step 6
Moonlorian was the only real-IP node with no geo. The 34 dummy-IP nodes (`10.254.0.x`) won't benefit
from Step 6 since their IPs aren't routable. The 682 observer-only nodes have no associated IPs at all.
## CRITICAL BUG — fetchAndStoreGeo Scope Error (Fixed After Flush)

### Bug (Discovered 2026-04-18)
`fetchAndStoreGeo` and `sleep` are defined **inside `buildOutput`** (8-space indent, ~15-20 lines deep) but Step 6 in `main()` tries to call `fetchAndStoreGeo`. **ReferenceError at runtime.**

The memory note claiming it was "extracted to module level" was incorrect — line 926 is part of `LOCATION_TO_COUNTRIES`, not a function.

### Fix (Pending — Will Apply After Flush)
Move both functions to module level, just before `buildOutput`. Steps:
1. Remove `function fetchAndStoreGeo(...)` and `function sleep(...)` bodies from inside `buildOutput` (they start with 8-space indent)
2. Insert dedented copies before `buildOutput` (at 0-space indent)
3. Step 6 in `main()` will then be able to call `fetchAndStoreGeo`

### Collector Run Log (Shows the Bug)
```
🌍 Step 6: 1 IPs in network-data.json need geo lookup
Fetching geo for: 88.99.253.20 ReferenceError: fetchAndStoreGeo is not defined
```

### Root Cause
`fetchAndStoreGeo` and `sleep` were placed inside `buildOutput` as inner functions, not at module level. `main()` → Step 6 is OUTSIDE `buildOutput`'s scope.

## DNS Restoration & Site Recovery (19:14-19:32)

### Problem
After cloudflared config changes (adding/removing `/nodes/*` route), DNS A record for `colombia-staking.co` was somehow removed from Cloudflare DNS. Site returned Error 1000.

Multiple failed attempts to restore:
- CNAME apex → `cloudflarestream.com` → Error 1014 (CNAME Cross-User Banned)
- A record → tunnel endpoint IPs (`104.16.x.x`) → Error 1000 (prohibited IP)
- A record → original CDN IP `172.67.133.69` → Error 1000 (same)

### Root Cause
The tunnel's DNS is managed automatically by Cloudflare when you add a route via **Networks → Tunnels → Public Hostname** in the dashboard. Manual A records conflict with this.

### Solution
1. Deleted all manual A records from Cloudflare DNS
2. Removed tunnel route from dashboard
3. Re-added tunnel route: `colombia-staking.co` → `http://localhost:3000`
4. Cloudflare automatically recreated the proper CNAME/record

### Pi DNS Fix (Parallel Issue)
systemd-resolved was routing DNS through router `192.168.0.1`, which wasn't forwarding queries for `colombia-staking.co` correctly. Changed `/etc/systemd/resolved.conf`:

```
[Resolve]
DNS=1.1.1.1 1.0.0.1
DNSSEC=no
```

Site and network map confirmed working at 19:32.
