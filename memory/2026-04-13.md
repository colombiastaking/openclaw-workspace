## Network Explorer - Sigma/Leaflet Geo Integration Fixes (evening session)

After the STEP F work (morning), a second debugging session tackled why sigma nodes weren't showing on the Leaflet geo map despite geo data loading correctly.

### Root Causes Found & Fixed

**1. Invalid sigma v3 constructor options** (commit `b45be9c`)
- `mouseWheelSmoothing`, `minCameraAngle`, `maxCameraAngle` are NOT valid sigma v3 options
- These caused the Sigma constructor to throw silently, leaving canvas blank
- Fix: removed all three options

**2. Leaflet double-init guard** (commit `b45be9c`)
- `_doInit()` could fire twice (once from `_bindEvents()` + once from elsewhere)
- Each call created a new Leaflet map, breaking the existing one
- Fix: `if (!this._geoMap)` guard in `_initLeafletMap()`

**3. `/api/geo` route order in kepler-proxy** (commit `f2871f2`)
- The `/api/*` catch-all was matching `/api/geo` before the specific `/api/geo` handler
- Result: 402 error from backend (no credits) — geo data was empty
- Fix: moved `/api/geo` route BEFORE the catch-all proxy route

**4. better-sqlite3 path in kepler-proxy** (commit `f2871f2`)
- `require('better-sqlite3')` was failing because kepler-proxy runs from `~/.openclaw/kepler/` but the module wasn't installed there
- Fix: absolute path `/home/raspberry/.openclaw/workspace/network-explorer/node_modules/better-sqlite3`

**5. `cached` variable hoisting** (commit `f2871f2`)
- `let cached` declared AFTER first use → JS Temporal Dead Zone error
- Fix: moved declaration to top of function

### Still Outstanding — Sigma/Leaflet Sync Conflict

The sigma camera and Leaflet map are fighting each other:
- Sigma `updated` event → drives Leaflet `setView()`
- Leaflet `move` event → drives sigma camera
- The `_skipGeoSync` / `_skipSigmaSync` flags exist but may not fully prevent the loop
- Zoom direction appears inverted (scroll up → zoom out instead of in)

**Suspected cause:** sigma v3 camera `ratio` → Leaflet zoom mapping has wrong sign/inversion
- Current: `zoom = cam.ratio * 2.5` (direct proportion)
- Likely needs: `zoom = Math.log2(cam.ratio)` or similar inverse relationship

**Also suspected:** sigma zoom events (wheel) fire `updated` → drive Leaflet → drive sigma back → loop

**What was NOT the issue (ruled out):**
- `_geoMap` was correctly initialized and tiles were loading fine
- Geo data was loading fine from `/api/geo`
- Node positions were correctly set in sigma (lat/lng → sigma coords)
- `_hideLoading()` was only called after ForceAtlas2 settled (600 ticks), but geo nodes shouldn't need FA

## Network Explorer - Sigma Visibility Fix (8:25 PM session)

**Problem:** Graph showed loading spinner but never transitioned to visible nodes.

**Root cause — two bugs from later commits:**

1. **`autoRescale: false`** — Added in commit `9cb54cf` after the last working version `167ac1a`. sigma v2 defaults to `autoRescale: true` (computes bounding box from node positions for rendering). Setting it to `false` broke sigma's internal size calculations → all nodes invisible.

2. **Wrong camera coordinates** — Camera was at `x=360, y=360, ratio=0.15` instead of `x=690, y=690, ratio=0.55` (the exact 167ac1a values). With cluster grid spanning 4380×4380 units, x=360 was way off-center.

**Fix applied (match 167ac1a exactly):**
- Removed `autoRescale: false`, `autoCenter: false`, `minCameraRatio`, `maxCameraRatio` from Sigma constructor
- Camera: `cam.x = 690; cam.y = 690; cam.ratio = 0.55`
- Constructor options now match 167ac1a's exact set

**Commit:** `9d9c8a4` — "Revert sigma constructor and camera to 167ac1a state"

**Status:** Pushed to GitHub. Graph should load with ForceAtlas2 spreading nodes around cluster centers, then pan/zoom works via sigma v2's built-in MouseCaptor.
