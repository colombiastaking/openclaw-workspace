# 2026-04-10

## Network Explorer — Full Public IP Map (Major Milestone)

**What changed today:**
The MultiversX Network Explorer now maps every validator and observer's **public IP address**, not just BLS identity. Each node with a resolvable public IP gets a grey diamond anchor at that IP, with a red edge connecting it to its IP. Nodes sharing the same public IP cluster around the same grey diamond.

**Key files modified:**
- `network-explorer/server/collector.js` — Complete rewrite of `buildOutput()`:
  - IP handling: Extracts `observedListeningAddr` from peer info, normalizes (strips `/tcp/` suffix), validates public IPs via `isLanIp()` check
  - Maps `peerBls → publicIP` (node-level) and `publicIP → [peerBls]` (anchor-level)
  - Builds `nodes[]` with `publicIp` field + `ipAnchor` boolean flag
  - Builds `edges[]` with `peerBls` (source node) + `peerBls` (target = anchor's BLS from `ipAnchorPeers`)
  - Stats: `publicIpNodes` (with public IP), `lanNodes` (LAN-only), `vpsNodes` (anchor nodes), `colombiaAtPi` (at Colombia's anchor IP)
  - Skips: shard key blocks, metachain key blocks, relay nodes, any peer with no `peerBls`
- `network-explorer/public/app.js` — Edge rendering update:
  - Reads `edge.peerBls` (always set, the node's BLS) instead of `edge.pk` (always empty)
  - Looks up `ipOf(edge.peerBls)` to get the node's public IP → `ipToCoords[ip]` for edge endpoints
  - Fallback: `node.x/y` if IP not in coordinate map
  - Removes stale `edge.pk` references throughout

**Results after deploy (live at colombia-staking.co/network/):**
- **6,198 nodes** — all MultiversX validators + observers with public IPs
- **1,378 edges** — one red line per node→IP pair
- **538 unique public IP anchors** — grey diamond points
- **59 Colombia Staking nodes** clustered around anchor at `186.99.3.198`
- **0 LAN IPs** — no 127.x, 192.168.x, 10.x, 172.x leaked into the map
- ⚪ White circles = regular nodes | 🟠 Orange = Colombia Staking | ⚫ Grey diamonds = public IP anchors | 🔴 Red lines = node→IP edges

**Remaining issue:** 3 edges with LAN IPs still visible (IPs `178.104.4.219`, `143.198.96.171`, `82.66.243.38`) — these appear to be nodes whose `observedListeningAddr` contains a LAN IP despite being VPS nodes. Likely a MultiversX API artifact where the node reports its internal LAN IP in the peer list. `isLanIp()` check should filter these but apparently doesn't catch all cases. Investigation ongoing.

**Git commits:**
- `colombiastaking/network-explorer`: pushed commit with full collector.js rewrite + app.js edge rendering fix
- `colombiastaking/alice-backup`: backed up latest state


## Network Explorer — Multikey Blue Edges (16:05-16:12 GMT-5)

**Feature added:** Blue edges for multikey IP assignments

**Problem:** TrustStaking group (100 nodes) — 4 observers had red edges (real peer connections), 96 validators appeared at IP `37.27.139.71` but with no edges (they were never directly observed as peers).

**Solution:** Added a second edge type that traces multikey IP assignments:
1. `collector.js` STEP B now also calls `addMultikeyEdges()` — for every identity group with exactly N provided IPs (e.g., 4 observers), distributes those IPs to all N members and creates blue edges
2. All 96 TrustStaking validators now have **blue edges** connecting them to their shared IP anchor
3. `public/app.js` updated: blue edges rendered in `#4da6ff` (light blue), line width 0.8, opacity 0.6 — visually distinct from red real peer edges

**Final edge breakdown:**
- 🔴 **Red** (1319 edges): directly observed P2P peer connections — only nodes actually seen as peers by our 8 observers
- 🔵 **Blue** (389 edges): multikey IP propagation — identity groups sharing IPs via multikey
- ⚫ **Grey** (4 edges): synthetic Colombia NAT edges — for nodes behind NAT without real peer data

**Git:** Commits pushed to `colombiastaking/network-explorer`
